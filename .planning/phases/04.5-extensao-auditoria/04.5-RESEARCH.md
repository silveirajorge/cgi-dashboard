## Summary

Esta fase estende o formulário de avaliação (Fase 4.2) e o dashboard de grupo (Fase 4.3) com novos campos de auditoria: 4 checkboxes booleanos (atraso, falta, uso_ferramenta, erro_critico), 2 campos numéricos (produtividade %, nota_auditoria 0-100) e tipo_auditoria (supervisor/auditor em semanas intercaladas). Requer migração SQLite com ALTER TABLE, adição de componentes shadcn/ui (Checkbox, RadioGroup), e extensão da API de stats com novas agregações.

**⚠️ Descoberta crítica — conflito de nomenclatura:** O nome `produtividade` no D4.5-01 COLIDE com a coluna existente `produtividade INTEGER` na tabela `avaliacoes`. A coluna existente é uma das 6 categorias de avaliação (escala 1-10). A nova coluna representa percentual de produtividade (REAL, pode exceder 100). **Necessário renomear a nova coluna** para `perc_produtividade` ou similar.

**Primary recommendation:** Seguir o padrão existente do código — formulário com estado local (não react-hook-form), migração SQLite com try-catch, e Recharts BarChart com múltiplos `<Bar>` para o gráfico comparativo.

## ⚠️ Critical Issue: Column Name Conflict

**Descoberta durante a pesquisa — precisa de decisão antes do plano.**

A decisão D4.5-01 define uma nova coluna `produtividade REAL`, mas a tabela `avaliacoes` JÁ possui uma coluna `produtividade INTEGER NOT NULL CHECK(produtividade >= 1 AND produtividade <= 10)` — uma das 6 categorias de avaliação (escala 1-10).

**Impacto:** SQLite lançará `duplicate column name: produtividade` se tentar executar `ALTER TABLE avaliacoes ADD COLUMN produtividade REAL`.

**Recomendação:** Renomear a nova coluna para `perc_produtividade` (produtividade percentual). Isto:
- Mantém semântica clara (a existente é categoria 1-10, a nova é %)
- Evita ambiguidade no formulário (ambas apareceriam como "Produtividade")
- Permite que a validação `CHECK(produtividade >= 1 AND produtividade <= 10)` existente permaneça intacta

**Ação necessária:** O plano deve usar `perc_produtividade` em vez de `produtividade` para a nova coluna, e ajustar todos os pontos de integração (POST body, SELECT, stats, props dos componentes).

### Pattern 1: SQLite Migration com try-catch

**What:** Adicionar colunas à tabela existente sem quebrar se já existirem. SQLite não suporta `IF NOT EXISTS` para `ALTER TABLE ADD COLUMN`.

**When to use:** Sempre que adicionar colunas a uma tabela SQLite existente.

**Example:**
```typescript
// src/lib/db.ts — padrão existente no código (linhas 39-43)
const migrations = [
  `ALTER TABLE avaliacoes ADD COLUMN atraso INTEGER NOT NULL DEFAULT 0`,
  `ALTER TABLE avaliacoes ADD COLUMN falta INTEGER NOT NULL DEFAULT 0`,
  `ALTER TABLE avaliacoes ADD COLUMN uso_ferramenta INTEGER NOT NULL DEFAULT 0`,
  `ALTER TABLE avaliacoes ADD COLUMN erro_critico INTEGER NOT NULL DEFAULT 0`,
  `ALTER TABLE avaliacoes ADD COLUMN perc_produtividade REAL`,
  `ALTER TABLE avaliacoes ADD COLUMN nota_auditoria REAL`,
  `ALTER TABLE avaliacoes ADD COLUMN tipo_auditoria TEXT CHECK(tipo_auditoria IN ('supervisor', 'auditor'))`,
];

for (const sql of migrations) {
  try {
    database.exec(sql);
  } catch {
    // Coluna já existe — ignorar
  }
}
```

### Anti-Patterns to Avoid

- **Criar componente Field:** O projeto não usa o padrão shadcn/ui Field. O formulário existente usa Label + Input diretamente. Manter consistência.
- **Usar react-hook-form:** O formulário existente usa estado local simples. Adicionar react-hook-form para apenas 7 campos seria overkill e inconsistente.
- **Try-catch genérico sem array:** Agrupar todas as migrações em um único try-catch impediria que migrações posteriores executassem se uma anterior falhar. Usar loop com try-catch individual (padrão robusto).

## Common Pitfalls

### Pitfall 1: Nome de coluna duplicado `produtividade`

**What goes wrong:** `ALTER TABLE avaliacoes ADD COLUMN produtividade REAL` falha porque `produtividade` já existe como `INTEGER NOT NULL`.

**Why it happens:** A coluna existente `produtividade` (1-10) é uma das 6 categorias de avaliação. O D4.5-01 reusa o mesmo nome sem notar o conflito.

**How to avoid:** Renomear para `perc_produtividade` no schema, API props, form state, e UI labels.

**Warning signs:** Testar migration com SQLite mostra erro `duplicate column name`.

### Pitfall 2: `nota_auditoria` e `tipo_auditoria` sem validação

**What goes wrong:** Usuário submete formulário sem nota ou tipo (violando D4.5-03).

**Why it happens:** O formulário existente valida apenas campos obrigatórios básicos (funcionário, data, notas 1-10).

**How to avoid:** Adicionar validação client-side no `handleSubmit` (antes do fetch) + validação server-side no POST. Seguir o padrão existente de `setError`.

### Pitfall 3: SELECT sem as novas colunas no GET /api/avaliacoes

**What goes wrong:** O GET usa lista explícita de colunas. Se não adicionar as novas, o frontend nunca as recebe.

**Why it happens:** O SELECT statement é explícito (`SELECT a.id, a.data_avaliacao, a.pontualidade, ...`) — não usa `SELECT *`.

**How to avoid:** Adicionar os 7 novos campos ao SELECT statement no GET route. A rota `[id]` usa `SELECT a.*` e não precisa de alteração.

### Pitfall 4: Recharts grouped bar com domínio fixo 0-10

**What goes wrong:** Se o gráfico comparativo usar `YAxis domain={[0, 10]}` (copiado do comparative-chart.tsx), funciona porque nota_auditoria é 0-100 e média geral é 0-10 — escalas diferentes.

**Why it happens:** O comparative-chart.tsx usa `domain={[0, 10]}` porque as categorias são 1-10. O novo gráfico compara notas de auditoria (0-100? ou 0-10?).

**How to avoid:** Verificar a escala esperada. Se `nota_auditoria` é 0-100, o domínio do YAxis deve ser `[0, 100]`. Se é 0-10 (média), usar `[0, 10]`. Confirmar com usuário.

### Stats API — novas agregações

```typescript
// Media nota_auditoria
const mediaNotaRow = db.prepare(`
  SELECT AVG(a.nota_auditoria) as media_nota_auditoria
  FROM avaliacoes a JOIN funcionarios f ON f.id = a.funcionario_id ${where}
`).get(...params) as { media_nota_auditoria: number | null };

// Totais booleanos
const totaisRow = db.prepare(`
  SELECT
    SUM(a.atraso) as total_atrasos,
    SUM(a.falta) as total_faltas,
    SUM(a.erro_critico) as total_erros_criticos
  FROM avaliacoes a JOIN funcionarios f ON f.id = a.funcionario_id ${where}
`).get(...params) as { total_atrasos: number; total_faltas: number; total_erros_criticos: number };

// Media produtividade
const mediaProdRow = db.prepare(`
  SELECT AVG(a.perc_produtividade) as media_produtividade
  FROM avaliacoes a JOIN funcionarios f ON f.id = a.funcionario_id ${where}
`).get(...params) as { media_produtividade: number | null };

// Comparativo por tipo_auditoria
const comparativoAuditoria = db.prepare(`
  SELECT
    f.nome as funcionario_nome,
    AVG(CASE WHEN a.tipo_auditoria = 'supervisor' THEN a.nota_auditoria END) as media_supervisor,
    AVG(CASE WHEN a.tipo_auditoria = 'auditor' THEN a.nota_auditoria END) as media_auditor
  FROM avaliacoes a
  JOIN funcionarios f ON f.id = a.funcionario_id
  ${where}
  GROUP BY f.id, f.nome
  ORDER BY f.nome
`).all(...params);
```

   - Recommendation: Usar `perc_produtividade` como nome técnico (curto, semântico, sem colisão). A label na UI pode continuar "Produtividade (%)".

3. **O gráfico "comparativo supervisor vs auditor" deve mostrar NOTA_AUDITORIA por tipo ou MÉDIA GERAL por tipo?**
